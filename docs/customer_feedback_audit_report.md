# Audit Report: Customer Feedback Template Logic

## Executive Summary
We have performed a comprehensive audit of the Customer Feedback Template logic (`lib/builder/customer-feedback-template.ts`) and the SQL query builder (`lib/builder/sql-builder.ts`) against the production database schema.

**Result:** ✅ **Passed**
The logic for calculations, filtering, and data aggregation is sound and aligns correctly with the database schema and data distribution.

## Detailed Findings

### 1. Schema Validation
All fields referenced in the template exist in the `reports` table and match the defined data types.

- **Reports Table**: `id`, `reports` table exists.
- **Dimensions**: `branch`, `airline`, `main_category`, `sub_category`, `area`, `target_division`, `created_at`, `hub`, `flight_number`, `report_content`, `root_cause`, `action_taken`, `evidence_url` are valid.

### 2. Calculation Logic
The template primarily uses aggregation functions (`COUNT`, `COUNT_DISTINCT`) which are correctly implemented in the `sql-builder`.

- **KPIs**: 
  - `Total Reports`: `COUNT(id)` - Correct.
  - `Distinct Branches`: `COUNT(DISTINCT branch)` - Correct.
  - `Distinct Airlines`: `COUNT(DISTINCT airline)` - Correct.
  - `Compliments`: Filter `main_category = 'Compliment'` + `COUNT(id)` - Correct.

### 3. Filtering Logic (Landside vs CGO)
The logic to separate "Landside" (General/Passenger) from "CGO" (Cargo) relies on the `target_division` field.

- **Logic**:
  - **Landside**: `target_division NOT IN ['UQ']`
  - **CGO**: `target_division IN ['UQ']`
- **Verification**:
  - We verified that `target_division` has no `NULL` values in the database.
  - We verified that `UQ` matches 75 records (likely Cargo/Unit Q).
  - The split is exhaustive and mutually exclusive.

### 4. Area Filtering
The dashboards correctly filter by the `area` column constants:
- `'TERMINAL'` (285 records)
- `'APRON'` (208 records)
- `'GENERAL'` (58 records)

### 5. Date Handling
The generic `dateFilters` function correctly applies `gte` and `lte` operators on the `created_at` timestamp field, matching the input `dateFrom` and `dateTo`.

## Conclusion
The queries generated by the template are accurate and statistically correct based on the current data. No code changes are required.
